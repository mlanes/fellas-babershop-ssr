# Colors

The brand palette is fixed; the surface colors flip via the `data-theme`
attribute on `<html>`. Components consume the **CSS custom properties**
below — they don't hardcode hex.

> Source of truth: `assets/styles/_color-definitions.scss`,
> `assets/styles/_primitive-colors.scss`.

## Theme tokens

Surface and text tokens that flip per `data-theme="light|dark"`.

| Token                    | Light      | Dark       |
| ------------------------ | ---------- | ---------- |
| `--page-background`      | near-white | near-black |
| `--page-color`           | near-black | near-white |
| `--text-color-primary`   | near-black | near-white |
| `--text-color-secondary` | gray-700   | gray-300   |
| `--surface-background`   | white      | dark gray  |
| `--surface-border`       | gray-200   | gray-700   |

Toggle the **Theme** toolbar above (paintbrush icon) to see every
component swap palettes.

## Status

| Token                    | Use                 |
| ------------------------ | ------------------- |
| `--status-color-success` | confirmation states |
| `--status-color-error`   | error states        |

## When to add a new color

Almost never. If something needs a one-off color, prefer composing
existing tokens (e.g. `color-mix(in srgb, var(--text-color-primary) 10%, transparent)`)
over introducing a new variable. If a new brand-level color is genuinely
needed, add it to `_primitive-colors.scss`, then surface it through
`_color-definitions.scss` — never reference primitives directly from
components.
